This page is a Work In Progress

This whole project started when I wanted to find option price data to backtest an earnings strategy. SoFi seemed to be a prime target for short-strangles/iron condors, due to its implied move historically being significantly higher than its realized move, and being a high IV stock premiums looked attractive. This strategy needed to be put to the test. I consulted MarketChameleon, my favorite website for such data, but while it was possible to get some data about historical implied vs. realized moves, the actual numbers were locked behind a 99$/month paywall. I then tried to get historical option data using the yfinance python library, but to no avail, only current prices are available. The solution then seemed clear: create a program that would fetch option data after every market close and store it, which could then be consulted whenever needed. This was about 3 weeks before SoFi's upcoming earnings (May 1st 2023), and a deadline was set for 2 days before their earnings, as to not miss this earnings cycle.

  • This project would be split into 3 parts:
  • a program capable of fetching option data and storing it
  • a device running 24/7 physically storing the data, accessible from anywhere, which would run after each market close
  • a program capable of accessing the aforementionned device, which would parse the data and be responsible for displaying it in a presentable and concise manner

    • Above is the actual function that is used to fetch and store option data. It seems simple now, but trial and error was required to dial down the proper format, as well as figure out error handling, which will be talked about later.

    The first order of business was a function that would accept a stock symbol and save all the data necessary. The first decision to make was how to store said data. The original intent was to create one csv file per day, with each column's header being the expiry date, and the corresponding data being stored in a singular cell beneath the header cell. This was done with the belief that saving a singular file would save on storage and processing speed. However, during testing it was found that when trying to access the data, python couldn't unpack the data properly, displaying mess of text instead of a full table. The decision was then made to instead store each expiry date's data in its own file, one for calls and one for puts.
